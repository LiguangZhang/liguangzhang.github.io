  1. 虚拟化type2，asid 如何mapping，因为user app 一定有asid，user mode 下应该只需要一级映射？怎么处理 ， APP直接对应vmid？
    ARM的vmid如何定义

在虚拟机os中, user app 只需要对应asid, 虚拟机os 进程管理中需要asid 页表切换 tlb 刷新机制与asid 相关.
vmid 是相对于 kvm (VMM) 来说的, 虚拟机的GPA 转换为 HPA (host 物理地址), 这个过程也是有mmu参与的, 需要vmid 标识内存属于哪一个虚拟机os.
二级地址翻译 GPA->HPA 需要vmid的参与. 一级地址翻译 GVA->GPA 需要asid 参与.
vmid 在不同平台上概念都是一样的.

而对于iommu DMA remapping过程 , DMA的数据流需要经过iommu 翻译(相当于mmu的二级地址翻译在iommu 这里只有一次翻译的过程), 同时需要asid 和 vmid 参与. 因为只有一次翻译, 所以需要确定DMA的数据流是来自于虚拟机的哪个进程.

  1. hgeip/hgeie 这两个如何用？ 不是有授权，为啥还需要这个
    授权指的是什么?
    这两个作为 VMM 管理虚拟机的中断机制使用, 通过这两个寄存器可以让VMM 控制某个虚拟机是否能接收外部中断. 不建议VMM 直接控制虚拟机的 sip/sie 寄存器, 会扰乱虚拟机os 运行的独立性(虚拟机的VS-mode下的csr虽然可以被HS-mode访问修改, 但是不建议这样做, 如果HS-mode kvm给它改了, guest os上的运行的软件并不会知道外部改了它的csr, 会有一致性问题). 参考问题6, 这两个csr 在kvm中暂时未用到

  2. 定义了Hypervisor的store load ？
    这个是指什么? 如果是说的虚拟机os里的store, load, 这个如果GPA mmu 没有找到真实的HPA, 会触发 guest load/store fault, 由 kvm(VMM) 接管, kvm 会分析该store/load 指令, 读写的哪个地址, 目标寄存器是什么, kvm 处理把数据写入或者读取 或者转发给qemu 处理, 再回到虚拟机中.

  3. qemu+kvm，其中qemu运行在那一层？ qemu+linux 运行在VS等级上
    qemu 运行在U-mode (也叫HU), linux kvm 运行在HS, 虚拟机的linux 运行在VS, 虚拟机的用户态运行在VU

下图的用户态虚拟机进程在这里就是qemu 进程

  1. 物理中断和虚拟中断，先发送物理中断，然后再配置对应的虚拟中断，有可能没有对应的虚拟中断，则需要hypervisor直接注入
    分两层:
    如果当前物理cpu 处在guest os 中, 一般是虚拟中断直接由虚拟cpu 接收
    如果当前物理cpu 处在 host os 中, 则判断中断是否要发给虚拟cpu, qemu 需要模拟中断控制器发给kvm, kvm 通过hvip 注入中断给虚拟机, 会涉及到host os -> guest os的上下文切换

  2. hypervisor下有virtual interrupt pending(hvip), hypervisor interrupt pending(hip), hypervisor guest external intr (hgeip),
    guest external intr 是对virtual machine （hip对应sip），hvip看起来时hypervisor要写一个虚拟中断

    是的, hvip 是向guest 注入虚拟中断用的, 目前kvm 中只看到用到了hvip, 其他几个暂未提及

  3. 一个physical的CPU上可以跑很多个虚拟机，每个虚拟机有很多guest os

问题里的虚拟机是指kvm(VMM) 吗, 一个host os 上只有一个kvm 模块. 一个物理cpu上只能跑一个kvm(VMM). kvm 可以托管多个guest os (VM)
一个物理cpu 只能绑定一个vcpu(只有一套vcpu csr) , 而vcpu 不同的ontext 可以对应不同的guest os.
物理cpu 上跑不同的guest os, 是分时的, 只要切换vcpu的 csr context 即可切换guest os.
